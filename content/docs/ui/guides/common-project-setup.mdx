This guide documents the standard setup used in our projects, covering framework configuration, styling, components, and tooling.

## Framework (Next.js 16)

We use Next.js as our React framework. To install a new Next.js project, follow the [official installation guide](https://nextjs.org/docs/app/getting-started/installation).

After installation, we use this configuration:

```tsx title="next.config.ts"
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  reactStrictMode: true,
  reactCompiler: true,
  typedRoutes: true,
  turbopack: {
    rules: {
      "*.svg": {
        loaders: ["@svgr/webpack"],
        as: "*.js",
      },
    },
  },
};

export default nextConfig;
```

- **`reactStrictMode`** - Enables additional development checks to find bugs early.
- **`reactCompiler`** - Automatically optimizes re-renders without manual `useMemo` or `useCallback`.
- **`typedRoutes`** - Provides TypeScript autocompletion and validation for route paths.
- **`turbopack`** - Next.js's faster Rust-based bundler, configured to handle SVG imports as React components.

For deployments we usually use [Vercel](https://vercel.com/docs). For internal projects, we also run a self-hosted Raspberry Pi server in the office.

## Tailwind CSS

Tailwind CSS is a utility-first CSS framework that lets you style elements directly in your markup. We use Tailwind CSS 4, which is a complete rewrite with CSS-first configuration.

For merging Tailwind classes, we use the `cn()` utility located in `src/lib/utils.ts`.

For theming, we use OKLCH color space instead of HSL for better perceptual uniformity. Colors are defined as CSS variables with light (`:root`) and dark (`.dark`) mode variants.

### Theme Switching

<Steps>
<Step>
**Install next-themes**

```bash
npm install next-themes
```
</Step>

<Step>
**Create a Providers component**

```tsx title="src/components/layout/providers.tsx"
"use client";

import { ThemeProvider } from "next-themes";

export function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider
      attribute="class"
      defaultTheme="system"
      enableSystem
      disableTransitionOnChange
    >
      {children}
    </ThemeProvider>
  );
}
```
</Step>

<Step>
**Wrap your app in the root layout**

```tsx title="app/layout.tsx"
import { Providers } from "@/components/layout/providers";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" suppressHydrationWarning>
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}
```
</Step>

<Step>

See the [Theme Toggle](/docs/ui/components/theme-toggle) component for a ready-to-use switcher.
</Step>
</Steps>

### Spacing Rules

Use `margin-top` for vertical spacing between sections. Inside flex/grid containers, use `gap` utilities instead.

```tsx
// Sections with margin-top
<section>First section</section>
<section className="mt-8">Second section</section>

// Flex/grid with gap
<div className="flex flex-col gap-4">
  <section>First</section>
  <section>Second</section>
</div>
```

### Sizing Rules

Always use `size-*` when width and height are equal:

```tsx
// ✅ Good
<SomeIcon className="size-4" />

// ❌ Bad
<SomeIcon className="w-4 h-4" />
```

## Component System (shadcn/ui)

We use [shadcn/ui](https://ui.shadcn.com) components built on Radix UI primitives. They provide full accessibility, are customizable via CSS variables, and use Class Variance Authority (CVA) for variants.

- [shadcn/ui Components](https://ui.shadcn.com/docs/components) - Full component library
- [Shadcn Blocks](https://www.shadcnblocks.com/blocks) - Pre-made blocks built with shadcn components

## Icons

We use [Lucide Icons](https://lucide.dev/icons/) for general icons, and our custom [Brand Icons](https://brands-gilt.vercel.app/) library for brand logos.

- All decorative icons **must** have `aria-hidden="true"`
- Use `size-*` utility for consistent sizing
- Only omit `aria-hidden` for icons with semantic meaning

## Forms

We use TanStack Form for form state management and Zod for validation.

### Field Component Anatomy

Use our custom Field components from `@/components/ui/field` for consistent form layouts:

```tsx
<Field orientation="vertical" data-invalid={isInvalid}>
  <FieldLabel htmlFor="email">Email *</FieldLabel>
  <Input {...inputProps} />
  <FieldDescription>Helper text here</FieldDescription>
  <FieldError errors={errors} />
</Field>
```

### Validation Patterns

```typescript
// Email
email: z.string().email("Please enter a valid email address.")

// Phone
phone: z.string()
  .min(9, "Phone number must be at least 9 characters.")
  .regex(/^[+]?[0-9\s\-()]+$/, "Please enter a valid phone number.")

// Required checkbox
consent: z.boolean().refine(val => val === true, {
  message: "You must agree to continue.",
})
```

## Images

We use `next/image` for rendering images, which provides automatic optimization, lazy loading, and responsive sizing.

For local/static images, we use our custom `StaticImage` component that wraps Next.js Image and automatically extracts dimensions from imported static image data.

```tsx title="src/components/ui/static-image.tsx"
import type { StaticImageData, ImageProps } from "next/image";
import Image from "next/image";

type StaticImageProps = Omit<ImageProps, "src"> & {
  image: StaticImageData;
};

export function StaticImage({ image, alt, ...props }: StaticImageProps) {
  return <Image src={image.src} alt={alt} width={image.width} height={image.height} {...props} />;
}
```

### SVG as React Components

SVGs are configured to import as React components via @svgr/webpack:

```tsx
import Logo from "@/assets/svgs/logo.svg";

<Logo className="size-8" aria-hidden="true" />
```

### Import Paths

- Use `@/` alias for src imports (e.g., `@/components/ui/button`)
- Use relative imports for same directory (e.g., `./helper-component`)

### Custom Link Component

Always use `Link` from `@/components/ui/link` instead of `next/link` directly. It provides type-safe routing with `typedRoutes: true`. See the [Link component](/docs/ui/components/link) for installation and usage.

## Configuration Management

The `config/` folder is the **single source of truth** for app configuration. Store site metadata, contact info, navigation links, and other settings here instead of hardcoding across components.

```tsx
import { site } from "@/config/site";
import { contact } from "@/config/contact";
import { navLinks } from "@/config/nav-links";
```

## Accessibility

Common ARIA attributes:

- **`aria-hidden="true"`** - Hides decorative elements from screen readers.
- **`aria-invalid`** - Tells screen readers the input has a validation error.
- **`aria-describedby`** - Links an element to its description/error message by ID.
- **`role="alert"`** - Makes content announce immediately when it appears.

```tsx
<Input
  id="email"
  aria-invalid={hasError}
  aria-describedby={hasError ? "email-error" : undefined}
/>
{hasError && <span id="email-error" role="alert">Invalid email</span>}
```

## Naming Conventions

| Type | Convention | Example |
|------|------------|---------|
| Components | PascalCase | `PrimaryButton` |
| Files | kebab-case | `primary-button.tsx` |
| Folders | kebab-case | `contact-form/` |
